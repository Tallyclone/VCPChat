# VChatSyncAdapter 使用说明（通俗版）

这份文档讲怎么使用 `VChatSyncAdapter`。它是装在 VCPChat 桌面端里的“同步助手”：负责看着本机 `AppData`，把聊天、配置、附件变化提交到同步中心，也负责把其他设备的变化写回本机 `AppData`。

## 一句话理解

- `VChatSyncAdapter` 放在 **VCPChat** 里运行。
- 它连接 VCPToolBox 里的 `VChatSyncCenter`。
- 它会扫描和监听本机 `VCPChat/AppData`。
- 它不会把 `history.json` 当文件镜像同步，而是推导消息级变更。
- 同步回来的内容会写回本机 AppData，所以 VChat 和 DeepMemo 还能按老方式读取。

## 你需要先准备什么

1. VCPToolBox 上已经启用了 `VChatSyncCenter`。
2. 你知道中心地址，例如：

```text
http://100.100.50.10:6005/api/plugins/VChatSyncCenter
```

3. 你知道中心同步密钥。
4. 本插件目录存在：

```text
G:\VCP\VCPChat\VCPDistributedServer\Plugin\VChatSyncAdapter
```

## 第 1 步：配置 config.env

Adapter 优先读取自己目录里的：

```text
G:\VCP\VCPChat\VCPDistributedServer\Plugin\VChatSyncAdapter\config.env
```

推荐先按下面方式配置：

```env
VCHAT_ADAPTER_ENABLED=true
VCHAT_ADAPTER_MODE=uninitialized
VCHAT_APPDATA_PATH=../AppData
VCHAT_SYNC_CENTER_URL=http://100.100.50.10:6005
VCHAT_SYNC_WS_URL=
VCHAT_SYNC_KEY=你的同步密钥

VCHAT_DEVICE_ID=pc-home
VCHAT_DEVICE_NAME=家里的电脑

VCHAT_WATCH_DEBOUNCE_MS=700
VCHAT_QUEUE_INTERVAL_MS=5000
VCHAT_PULL_INTERVAL_MS=15000
VCHAT_ATTACHMENT_MULTIPART_THRESHOLD_BYTES=8388608
VCHAT_SYNC_ENABLE_WS=true
VCHAT_BOOTSTRAP_MODE=manual
VCHAT_RELEASE_MODE=mvp-local-only
DebugMode=false
```

必须改的项：

| 配置项                  | 怎么填                                                                                 |
| ----------------------- | -------------------------------------------------------------------------------------- |
| `VCHAT_SYNC_CENTER_URL` | 填 VCPToolBox 宿主地址，推荐只写到端口；也兼容完整 `/api/plugins/VChatSyncCenter` 前缀 |
| `VCHAT_SYNC_WS_URL`     | 可选；通常留空自动推导，单独配置时也可只写到端口并自动补全 WebSocket 路径              |
| `VCHAT_SYNC_KEY`        | 必须和中心侧完全一致                                                                   |
| `VCHAT_DEVICE_ID`       | 每台设备必须唯一，例如 `pc-home`、`pc-office`                                          |

首次接入建议：

```env
VCHAT_ADAPTER_MODE=uninitialized
VCHAT_BOOTSTRAP_MODE=manual
```

这样不会一启动就自动导入或合并，比较安全。

## 第 2 步：按需配置 runtime 配置同步字段

如果你只同步默认安全字段，可以跳过本步骤。

高级配置文件：

```text
G:\VCP\VCPChat\VCPDistributedServer\Plugin\VChatSyncAdapter\config\sync_profile.json
```

默认结构建议包含 `deleteMissing`：

> 注意：Center 的 `/bootstrap/export` 会同时导出 `bootstrap` 与 `runtime` 两类配置 profile。`bootstrap` profile 是完整安全基线；`runtime` profile 会随基线导出一起下发，避免新设备首次加入后跳过历史运行时配置变更。主题同步也会随基线/变化一起处理：主题包走 `/themes`，主题资源走 `/themes/assets`，落地后会写回 `styles/themes/*.css` 与 `assets/wallpaper/*`。

```json
{
  "version": 1,
  "runtimeSync": {
    "agent_config": {
      "include": [],
      "exclude": [],
      "deleteMissing": false
    },
    "group_config": {
      "include": [],
      "exclude": [],
      "deleteMissing": false
    }
  },
  "bootstrap": {
    "conflictPolicy": "report_only"
  }
}
```

### Agent 默认会同步什么配置？

默认同步这些运行时字段：

```text
name
topics
advancedSystemPrompt.hiddenBlocks
advancedSystemPrompt.warehouseOrder
advancedSystemPrompt.viewMode
presetSystemPrompt
selectedPreset
```

不会默认同步完整提示词、模型、CSS、TTS 等字段。

### 我想同步模型或提示词怎么办？

可以在 `include` 里加字段，例如：

```json
{
  "runtimeSync": {
    "agent_config": {
      "include": ["systemPrompt", "model", "temperature"],
      "exclude": [],
      "deleteMissing": false
    }
  }
}
```

Center 不再额外硬拒绝 agent/group runtime 字段；能不能同步主要由本机 Adapter 的 `sync_profile.json` 决定。Center 仍会做底线校验：字段必须在 bootstrap allowlist 内，runtime 必须带 `projection_fields`，`deleted_fields` 必须落在 `projection_fields` 内，同步内容不能包含敏感 key，且实际字段必须落在 `projection_fields` 内。

### 旧设备缺字段会不会删掉新设备字段？

默认不会。

`deleteMissing` 默认是 `false`。如果旧设备没有某个字段，Adapter 不会把它写进 `projection_fields`，另一台设备就不会把该字段删掉。

只有你明确写：

```json
"deleteMissing": true
```

Adapter 才会把缺失字段写入 `deleted_fields`，由 Center 校验后转发；接收端只删除 `deleted_fields` 中明确声明的字段。未进入 `deleted_fields` 的缺失字段不会被删除。不确定时不要开启。

## 第 3 步：重启 VCPChat / VCPDistributedServer

修改 `config.env` 或 `sync_profile.json` 后，重启 VCPChat 或它的 VCPDistributedServer，让插件重新读取配置。

本地接口前缀通常是：

```text
http://127.0.0.1:<VCPChat或VCPDistributedServer端口>/api/vchat-sync-adapter
```

下面用变量表示：

```powershell
$Adapter = "http://127.0.0.1:5974/api/vchat-sync-adapter"
$Key = "258456"
$Headers = @{ Authorization = "Bearer $Key" }
```

> 端口取决于你本机 VCPChat/VCPDistributedServer 实际监听端口。请用项目当前启动日志或配置确认。

## 第 4 步：检查 Adapter 状态

```powershell
$Adapter = "http://127.0.0.1:5974/api/vchat-sync-adapter"
$Key = "258456"
$Headers = @{ Authorization = "Bearer $Key" }
Invoke-RestMethod -Uri "$Adapter/status" -Headers $Headers -Method Get
```

重点看：

| 字段            | 说明                    |
| --------------- | ----------------------- |
| `ok`            | 接口是否正常            |
| `enabled`       | Adapter 是否启用        |
| `mode`          | 当前同步模式            |
| `app_data_path` | 实际读取的 AppData 路径 |
| `sync_dir`      | 同步状态目录            |
| `device_id`     | 当前设备 ID             |
| `queue`         | 离线队列状态            |
| `pull`          | 拉取中心事件状态        |
| `index`         | 本地索引状态            |

如果 `mode` 还是 `uninitialized`，说明还没有完成首次接入。

## 第 5 步：选择首次接入方式

### 模式 A：bootstrap_primary

适合：第一台电脑，中心还是空的，你想把这台电脑当前 AppData 作为初始数据。

```powershell
Invoke-RestMethod -Uri "$Adapter/bootstrap/bootstrap_primary" -Headers $Headers -Method Post -ContentType "application/json" -Body "{}"
```

建议：执行前先备份本机 `AppData`，并确认中心是新库或空库。

### 模式 B：join_existing

适合：第二台或第三台电脑，中心已经有数据，你希望这台电脑加入同步。

```powershell
Invoke-RestMethod -Uri "$Adapter/bootstrap/join_existing" -Headers $Headers -Method Post -ContentType "application/json" -Body "{}"
```

### 模式 C：merge_existing

适合：这台电脑本地也有一些数据，中心也有数据，需要尝试合并。

```powershell
Invoke-RestMethod -Uri "$Adapter/bootstrap/merge_existing" -Headers $Headers -Method Post -ContentType "application/json" -Body "{}"
```

建议：这是最容易出现冲突的模式。执行前务必备份本机 `AppData` 和中心数据库。

## 第 6 步：Bootstrap 后再次检查状态

```powershell
Invoke-RestMethod -Uri "$Adapter/status" -Headers $Headers -Method Get
```

如果模式已经不再是 `uninitialized`，并且 queue/pull 没有明显错误，就可以正常使用。

## 日常同步是怎么发生的

1. 你在 VChat 里聊天、编辑、删除消息，或者改配置。
2. AppData 文件变化。
3. Adapter 监听到变化。
4. Adapter 等文件稳定后读取。
5. Adapter 推导出新增消息、修改消息、删除消息、配置变化、主题资源等 operation。
6. operation 进入 `offline_queue.jsonl`。
7. 网络正常时提交给 VChatSyncCenter。
8. 其他设备从中心拉到变化。
9. 其他设备的 Adapter 把变化写回自己的 AppData，以及 `styles/themes/*.css`、`assets/wallpaper/*`。
10. 其他设备的 VChat 和 DeepMemo 就能读到这些内容。
11. 其他设备的 VChat 和 DeepMemo 就能读到这些内容。

## 常用命令

### 查看状态

```powershell
Invoke-RestMethod -Uri "$Adapter/status" -Headers $Headers -Method Get
```

### 查看本机 manifest

```powershell
Invoke-RestMethod -Uri "$Adapter/bootstrap/manifest" -Headers $Headers -Method Get
```

### 扫描 normalized_id 冲突

```powershell
Invoke-RestMethod -Uri "$Adapter/bootstrap/conflicts" -Headers $Headers -Method Get
```

### 第一台电脑初始化中心

```powershell
Invoke-RestMethod -Uri "$Adapter/bootstrap/bootstrap_primary" -Headers $Headers -Method Post -ContentType "application/json" -Body "{}"
```

### 新电脑加入中心

```powershell
Invoke-RestMethod -Uri "$Adapter/bootstrap/join_existing" -Headers $Headers -Method Post -ContentType "application/json" -Body "{}"
```

### 本机和中心合并

```powershell
Invoke-RestMethod -Uri "$Adapter/bootstrap/merge_existing" -Headers $Headers -Method Post -ContentType "application/json" -Body "{}"
```

### 修复某个话题里“本机有、中心没有”的消息

适合场景：你发现某个聊天在这台电脑本地能看到消息，但其他设备收不到，怀疑这些消息没有成功上传到中心。

这个修复命令只处理**一个话题**。它会读取本机对应的 `history.json`，再去中心检查同一批消息是否存在：

- 本机有、中心没有：会准备补发。
- 父 topic 在中心也没有：会先补父 topic，再补消息。
- 中心已经有的消息：不会因为本机索引缺失就重复补发。
- 默认 `dryRun=true` 只检查不修改，确认结果后再执行 `dryRun=false`。

先 dry run 看看会修什么：

```powershell
$Body = @{
  item_type = "group"
  item_id = "code____1778463518060"
  topic_id = "group_topic_1780747986280"
  dryRun = $true
} | ConvertTo-Json

Invoke-RestMethod -Uri "$Adapter/repair/topic" -Headers $Headers -Method Post -ContentType "application/json" -Body $Body
```

如果返回里的 `to_enqueue` 正是你要补的消息，再真正执行修复：

```powershell
$Body = @{
  item_type = "group"
  item_id = "code____1778463518060"
  topic_id = "group_topic_1780747986280"
  dryRun = $false
} | ConvertTo-Json

Invoke-RestMethod -Uri "$Adapter/repair/topic" -Headers $Headers -Method Post -ContentType "application/json" -Body $Body
```

字段怎么填：

| 字段        | 怎么理解                                                  | 例子                        |
| ----------- | --------------------------------------------------------- | --------------------------- |
| `item_type` | 聊天对象类型。群聊填 `group`，单 Agent 聊天通常填 `agent` | `group`                     |
| `item_id`   | 聊天对象 ID，也是 `AppData/UserData/` 下面那层目录名      | `code____1778463518060`     |
| `topic_id`  | 话题 ID，也是 `topics/` 下面那层目录名                    | `group_topic_1780747986280` |
| `dryRun`    | `true` 只检查不入队；`false` 才把修复操作写入离线队列     | `true` / `false`            |

如果你不知道这三个 ID，可以从本机历史文件路径里看出来。例如：

```text
AppData/UserData/code____1778463518060/topics/group_topic_1780747986280/history.json
```

对应就是：

```text
item_id  = code____1778463518060
topic_id = group_topic_1780747986280
```

`item_type` 看这个话题属于群聊还是单 Agent：群聊填 `group`，单 Agent 填 `agent`。

返回结果重点看这些字段：

| 返回字段                 | 含义                                                     |
| ------------------------ | -------------------------------------------------------- |
| `local_count`            | 本机这个 `history.json` 里有多少条有效消息               |
| `center_topic_exists`    | 中心是否已有父 topic；如果是 `false`，修复时会先补 topic |
| `missing_in_center`      | 本机有、中心没有的消息 ID                                |
| `missing_in_local_index` | 本机索引里缺失的消息 ID，仅用于辅助判断                  |
| `checksum_conflicts`     | 中心已有同 ID 但内容校验不同的消息，需人工确认           |
| `to_enqueue`             | 本次准备写入离线队列、等待上传中心的修复项               |
| `enqueued_count`         | `dryRun=false` 时实际写入离线队列的数量                  |

修复后不是直接写中心数据库，而是写入本机 `offline_queue.jsonl`，由 Adapter 按正常同步流程上传到中心。执行后可以再查一次状态：

```powershell
Invoke-RestMethod -Uri "$Adapter/status" -Headers $Headers -Method Get
```

也可以稍等几秒后再 dry run 一次同一个话题。如果 `missing_in_center` 和 `to_enqueue` 变空，通常说明已经补上。

注意：

- 建议先 `dryRun=true`，确认结果合理后再 `dryRun=false`。
- 如果 `checksum_conflicts` 不为空，不要盲目反复修复，先人工确认哪边内容才是正确的。
- 当前命令一次只修一个 topic；多个话题要分别执行。
- 修复命令仍需要 `$Headers` 鉴权，密钥就是 `VCHAT_SYNC_KEY`。

## 同步状态文件在哪里

Adapter 会在 AppData 下维护同步状态：

```text
VCPChat/AppData/sync/
├── state.json             # 当前模式、游标、最近扫描结果等
├── local_index.json       # 本地消息/文件索引
├── offline_queue.jsonl    # 离线待提交队列
└── write_intents.jsonl    # 防回环写入意图记录
```

一般不需要手动改这些文件。不要随便删除 `offline_queue.jsonl`，里面可能有未提交操作。

## 两台电脑同步的推荐步骤

### 第一台电脑

1. 配好中心 `VChatSyncCenter`。
2. 第一台电脑配置唯一 `VCHAT_DEVICE_ID`，例如 `pc-home`。
3. 重启 VCPChat。
4. 检查状态。
5. 执行 `bootstrap_primary`。

### 第二台电脑

1. 第二台电脑配置唯一 `VCHAT_DEVICE_ID`，例如 `pc-office`。
2. `VCHAT_SYNC_CENTER_URL` 和 `VCHAT_SYNC_KEY` 与第一台一致。
3. 重启 VCPChat。
4. 执行 `join_existing`。
5. 检查状态正常后，在任意一台电脑聊天，观察另一台是否同步。

## 和中心插件的联调命令

```powershell
$Center = "http://100.100.50.10:6005/api/plugins/VChatSyncCenter"
Invoke-RestMethod -Uri "$Center/status" -Headers $Headers -Method Get
```

查看中心变更：

```powershell
Invoke-RestMethod -Uri "$Center/changes?after_seq=0&limit=20" -Headers $Headers -Method Get
```

如果中心有变更，但本机不更新，重点查 Adapter 的 `pull` 状态。

如果本机有变化，但中心没有变更，重点查 Adapter 的 `queue` 状态和中心地址/密钥。

## 常见问题

### 1. 401 / adapter bootstrap authorization failed

同步密钥不对。检查：

- `config.env` 的 `VCHAT_SYNC_KEY`。
- 命令里的 `$Key`。
- 中心侧 `VCHAT_SYNC_KEY`。

三处必须完全一致。

### 2. status 显示 enabled=false

检查：

```env
VCHAT_ADAPTER_ENABLED=true
```

然后重启 VCPChat。

### 3. mode 一直是 uninitialized

说明还没完成 bootstrap。按你的场景执行：

- 第一台：`bootstrap_primary`
- 新加入：`join_existing`
- 双方都有数据：`merge_existing`

### 4. 本机改了聊天，但中心没变化

检查：

- `VCHAT_SYNC_CENTER_URL` 是否写对。
- `VCHAT_SYNC_KEY` 是否一致。
- `/status` 里的 queue 是否堆积。
- 当前 mode 是否允许上传。

`uninitialized` 模式下不会上传本机变更。

### 5. 另一台电脑没有收到变化

检查：

- 中心 `/changes` 是否能看到新事件。
- 另一台 Adapter 的 `/status` 里 pull 是否正常。
- 两台机器 `VCHAT_DEVICE_ID` 是否重复。

### 6. WebSocket 报错怎么办

WebSocket 只是加速通知，REST 轮询仍会同步。可以设置：

```env
VCHAT_SYNC_ENABLE_WS=false
```

重启后靠轮询同步。

### 7. systemPromptPresets 子目录里的 preset 会同步吗？

会。现在支持多层 JSON：

```text
systemPromptPresets/分类/子分类/preset.json
```

## 安全提醒

- 首次 bootstrap 前先备份 `VCPChat/AppData`。
- `merge_existing` 前一定备份，因为它最容易产生冲突。
- 不要手动编辑 `AppData/sync/offline_queue.jsonl`。
- 不要让两套 Adapter 同时操作同一个 AppData 目录。
- 每台设备必须使用不同的 `VCHAT_DEVICE_ID`。
- 不要把 API Key、token、Cookie、密码等敏感内容放进要同步的配置里。
- 不确定时不要开启 `deleteMissing: true`。
